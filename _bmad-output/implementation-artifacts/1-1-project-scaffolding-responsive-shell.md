## Story

As a developer,
I want to initialize the Next.js project with Tailwind, Shadcn, and a dessert-themed shell,
so that I have a modern, high-contrast foundation for the game.

## Acceptance Criteria

1. Given a new project directory, when the scaffolding commands are run, then the Next.js app starts successfully without errors.
2. A basic app layout shell is visible with a navigation bar.
3. A Dark/Light theme toggle is visible and functional.
4. Dessert-themed CSS variables are defined in `src/app/globals.css`.
5. The shell is mobile-first, high-contrast, and uses restrained Liquid Glass styling without harming readability.

## Dev Notes

### Story Scope

This story creates the application foundation only. Do not implement Connect Four rules, reducer behavior, bot logic, hints, coach logic, rewards, Supabase persistence, or leaderboard data beyond empty folders/placeholders required for structure. Later stories own those behaviors.

### Technical Requirements

- Framework: Next.js App Router web stack with React DOM, not Phaser/Pixi/canvas/game-engine architecture.
- Required runtime and libraries from project context:
  - Next.js App Router 16.2.6 target.
  - React 19.2.6 target.
  - Tailwind CSS 4.3.0 target.
  - Shadcn UI CLI 4.8.2 target.
  - `@supabase/supabase-js` 2.106.2 installed for future backend work.
  - `next-themes` installed for Dark/Light mode.
  - Node.js >=20.9.0.
- Preferred scaffold command for this existing repo:

```bash
npx create-next-app@latest . --ts --tailwind --app --src-dir --eslint --import-alias "@/*"
npx shadcn@latest init
npm install @supabase/supabase-js next-themes
```

If the CLI refuses to scaffold into a non-empty directory, use the documented `create-next-app` options in a temporary sibling directory and copy only the generated app/config files into this repository, preserving `_bmad`, `_bmad-output`, `.agents`, `docs`, and git metadata.

### Architecture Compliance

- The project is a web app and should render with React DOM/CSS components.
- The first screen should be an actual app shell for the game product, not a pure landing page.
- Keep the shell mobile-first and usable with one hand.
- Use Shadcn primitives for common buttons, cards, dialogs, tabs, and toggles as they are added.
- Do not block quick play behind auth. Auth, realtime multiplayer, Stripe, full city leaderboard, and Azure/LLM Coach are roadmap unless core MVP is complete.
- No Supabase write should happen in this story.
- Never log Supabase keys, tokens, PII, email addresses, or raw profiles.

Naming rules:

- React component files: `kebab-case.tsx`.
- Hooks: `use-kebab-name.ts`.
- Pure modules: `kebab-case.ts`.
- Tests: `<module>.test.ts`.
- Types/interfaces: `PascalCase`.
- Functions: `camelCase`.
- Reducer actions: `UPPER_SNAKE`.
- Log event names: `dot.case`.

### Styling and UX Guardrails

- Dessert-themed palette should feel warm and food-marketplace aligned, while avoiding a one-note beige/brown/orange UI.
- Board readability is more important than decoration. Even though the board is not implemented in this story, global tokens must support high-contrast future board/disc states.
- Liquid Glass belongs on shell, panels, dialogs, reward cards, and coach surfaces. Avoid stacked heavy blur and avoid blur over future board cells.
- Color must not be the only planned win indicator; token names/classes should leave room for ring/pattern/highlight treatment in later stories.
- Respect reduced motion where practical.
- Text must not overlap or overflow on mobile.

### Testing and Validation Requirements

- This story does not require game logic unit tests because no game logic is implemented.
- Required validation is scaffold-level:
  - lint/type validation command passes;
  - production build passes;
  - local app starts and renders the shell without runtime errors;
  - theme toggle visibly changes light/dark theme;
  - desktop and mobile viewport checks show no obvious layout overlap.
- If the scaffold uses a test runner by default, keep it configured. Do not add a new test framework unless the scaffold or project conventions already require it.

### Project Context Rules

- `src/lib/game/*` must stay pure TypeScript and must not import React, Next.js, Supabase, browser storage, DOM APIs, or UI components.
- UI may import game logic later, but must not duplicate rules.
- Supabase modules may persist summaries later, but must not mutate live match state.
- Components should dispatch reducer actions in later game stories instead of mutating board arrays directly.
- Bot, hint, and coach logic must eventually share tactical analysis helpers.
- Match completion must eventually follow: result -> coach insight -> reward -> summary persistence, with UI continuing if persistence fails.
- Expected game outcomes should use typed result objects later; async boundaries use try/catch.
- Commit only `.env.example`, never `.env.local`.

### Latest Technical Notes

- Official Next.js docs describe the App Router as a file-system router using current React features such as Server Components, Suspense, and Server Functions.
- `create-next-app@latest` supports `--ts`, `--tailwind`, `--eslint`, and `--app`; `--src-dir` places the app under `src/app` and configures the `@/*` alias accordingly.
- Tailwind CSS v4 usage in Next.js imports Tailwind from global CSS with `@import 'tailwindcss';`.
- Shadcn's current Next.js guide supports initializing an existing app with `npx shadcn@latest init`; the CLI init installs dependencies, adds the `cn` utility, and configures CSS variables.
- `next-themes` supports App Router usage and class/data-attribute theme switching; use a mounted client toggle pattern to avoid hydration mismatch.
